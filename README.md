# ☕ BLACK-AND-BREW ERP

> ระบบจัดการร้านกาแฟ **BLACK AND BREW** แบบครบวงจร — รวมตารางงาน คลังสินค้า ยอดขาย การซ่อมบำรุง และผู้ช่วย AI ชื่อ **บรู** ไว้ในเว็บแอปเดียว พนักงานเปิดจากมือถือหรือแท็บเล็ตในร้านได้ทันที ไม่ต้องสลับหลายโปรแกรม

**เวอร์ชัน 9.2** · รองรับภาษาไทย (หลัก) และอังกฤษ · เข้าเว็บแล้วไปหน้า `/th` อัตโนมัติ · ต้องใส่ PIN 6 หลักก่อนใช้งาน

## ✨ Features

> ทุกโมดูลออกแบบมาให้พนักงานหน้าร้านใช้งานบนมือถือได้จริง ไม่ใช่แดชบอร์ดสำเร็จรูปที่ต้องปรับเยอะ แต่ผูกกับงานร้านกาแฟโดยตรง — ตั้งแต่เช็กว่าวันนี้ใครมาทำงาน ไปจนถึงนับสต็อกและถาม AI เรื่องยอดขาย

### สรุปโมดูล

| หมวด | โมดูล | เส้นทางในเว็บ |
| --- | --- | --- |
| งานประจำวัน | Command Center, Staff Dashboard | `/[locale]`, `/[locale]/dashboard` |
| ตารางงาน | Schedule | `/[locale]/schedule` |
| คลังสินค้า | Inventory, Stock Count, Accuracy, Branch Withdraw | `/[locale]/inventory/*` |

> `[locale]` คือรหัสภาษา เช่น `th` (ไทย) หรือ `en` (อังกฤษ)

### 🏠 งานประจำวัน

#### Command Center

`/[locale]` · หน้าแรกหลังเข้าระบบสำเร็จ

**ใช้ทำอะไร:** เป็นหน้าสรุปสถานะร้านประจำวัน — ดูได้ในครั้งเดียวว่าวันนี้และพรุ่งนี้มีพนักงานคนไหนอยู่กะอะไร มีกี่คนประจำเคาน์เตอร์กาแฟ (หน้าร้าน) ใครลา ใครหยุด และกะไหนกำลังจะเริ่มหรือจบ

**เหมาะกับใคร:** หัวหน้ากะ ผู้จัดการ หรือพนักงานที่ต้องการเช็กภาพรวมทีมก่อนเปิดร้านหรือก่อนปิดร้าน

**ทำงานอย่างไร:**

- เมื่อมีคนแก้ตารางงาน หน้านี้อัปเดตทันทีโดยไม่ต้องรีเฟรช — ข้อมูลมาจากฐานข้อมูลแบบเรียลไทม์
- สีของการ์ดแต่ละกะขึ้นอยู่กับ **เวลาเริ่มงาน** (เช่น กะ 6:30 โทนสีเขียวอ่อน) ช่วยให้มองแล้วแยกกะได้เร็ว แม้ไม่จำชื่อกะ
- นับจำนวนพนักงานหน้าร้าน (FOH Coffee) แยกจากกะอื่น เพื่อรู้ว่ามีคนดูแลลูกค้าที่เคาน์เตอร์เพียงพอหรือไม่
- แสดงทั้งวันนี้และพรุ่งนี้ เตรียมตัวล่วงหน้าได้

**จุดเด่น:** แดชบอร์ดทั่วไปมักโชว์กราฟยอดขายหรือตัวเลขย้อนหลัง — หน้านี้ตอบคำถามที่สำคัญที่สุดตอนเปิดร้าน: *"วันนี้มีใครอยู่บ้าง และร้านพร้อมเปิดหรือยัง?"*

#### Staff Dashboard

`/[locale]/dashboard` · ลงเวลาและดูตารางรายเดือน

**ใช้ทำอะไร:** พื้นที่ส่วนตัวของพนักงาน — ดูกะของตัวเอง ลงเวลาเข้า-ออกงาน ดูรายชื่อคนที่อยู่กะเดียวกัน และเปิดดูตารางทั้งเดือนแบบปฏิทิน

**เหมาะกับใคร:** พนักงานทุกคนที่ต้องการเช็กตารางตัวเองหรือบันทึกเวลาทำงาน

**ทำงานอย่างไร:**

- โหลดข้อมูลกะครั้งเดียว แล้วแยกแสดงทั้งมุมมองรายสัปดาห์และรายเดือน — ลดเวลารอโหลด
- ข้อมูลกะมาจากแหล่งเดียวกับหน้า Schedule — ถ้าผู้จัดการแก้ตาราง หน้านี้จะตรงกันเสมอ
- แสดงรายชื่อกะวันนี้พร้อมสีตามเวลาเริ่มงาน เหมือนหน้า Command Center

**จุดเด่น:** ไม่ต้องใช้แอปลงเวลาแยกกับแอปดูตาราง — ทุกอย่างอยู่ในระบบเดียว ลดความสับสนเมื่อข้อมูลไม่ตรงกัน

### 📅 ตารางงาน

#### Schedule

`/[locale]/schedule` · จัดกะแบบลากวาง

**ใช้ทำอะไร:** ผู้จัดการลากวางกะพนักงานบนปฏิทิน กำหนดวันหยุด วันลา วันหยุดประจำ และดึงวันหยุดนักขัตฤกษ์ไทยมาแสดงอัตโนมัติ

**เหมาะกับใคร:** ผู้จัดการร้าน หัวหน้ากะ หรือผู้ที่มีสิทธิ์แก้ไขตาราง (PIN แบบเต็มสิทธิ์)

**ทำงานอย่างไร:**

- ลากการ์ดกะไปวางบนวันที่ต้องการ — ใช้นิ้วบนแท็บเล็ตได้
- วันหยุดราชการดึงจาก Google Calendar แล้วรวมกับวันหยุดที่ตั้งเองในระบบ
- เมื่อบันทึกแล้ว หน้า Command Center และ Dashboard ของพนักงานจะเห็นการเปลี่ยนแปลงทันที
- รองรับกะหลายแบบ เช่น กะเช้า กะบ่าย กะเคาน์เตอร์ หรือวันหยุด

**จุดเด่น:** สีกะไม่ได้ตายตัวตามตำแหน่งงาน แต่เปลี่ยนตามเวลาเริ่มกะ — สอดคล้องกับวิธีที่พนักงานร้านกาแฟมองตารางจริงๆ คือ "กะนี้เริ่มกี่โมง"

### 📦 คลังสินค้า

#### Inventory

`/[locale]/inventory` · ตารางคลังแบบ Excel

**ใช้ทำอะไร:** จัดการสินค้าในคลังทั้งหมดในตารางเดียว — แก้จำนวนสต็อก จุดที่ต้องสั่งซื้อเพิ่ม เป้าหมายสต็อกที่อยากเก็บไว้ หน่วยนับ และแหล่งที่มาของสินค้า นอกจากนี้ยังมีปุ่มลัดสำหรับรับ-จ่ายสินค้าเข้า-ออกคลังแบบเร็ว

**เหมาะกับใคร:** พนักงานที่นับสต็อก ผู้จัดการคลัง หรือคนที่รับของเข้า-เบิกของออกทุกวัน

**ทำงานอย่างไร:**

- แก้ตัวเลขได้ในช่องตารางเลย ไม่ต้องเปิดหน้าต่างย่อย — เหมือนใช้ Excel บนเว็บ
- เมื่อออกจากช่องหรือกด Enter ระบบบันทึกให้อัตโนมัติ
- การอัปเดตสต็อกทำผ่านฐานข้อมูลแบบปลอดภัย — ล็อกแถวก่อนแก้ ป้องกันสองคนแก้พร้อมกันแล้วข้อมูลเพี้ยน
- กดยกเลิกหรือทำซ้ำได้ในคลิกเดียว หากแก้ผิด
- หลายคนเปิดหน้าคลังพร้อมกันได้ — เห็นการเปลี่ยนแปลงของกันและกันแบบเรียลไทม์
- ถ้าเน็ตหลุด ยังแก้ไขบางอย่างได้ — พอกลับมาออนไลน์ระบบส่งข้อมูลขึ้นให้เอง
- เมื่อมีคนแก้สต็อก มือถือเครื่องอื่นที่เปิดระบบอยู่จะได้รับการแจ้งเตือน (ยกเว้นเครื่องที่เป็นคนแก้)

**จุดเด่น:** ระบบคลังทั่วไปมักบังคับกรอกทีละรายการผ่านฟอร์ม — ที่นี่ออกแบบให้นับสต็อกบนมือถือได้เร็ว ปลอดภัย และยังใช้ได้บางส่วนแม้ไม่มีเน็ต

#### Stock Count

`/[locale]/inventory/count` · ตรวจนับสต็อกจริง

**ใช้ทำอะไร:** ใช้ตอนตรวจนับสต็อกจริงในคลัง — พนักงานกรอกจำนวนที่นับได้ ระบบเทียบกับตัวเลขในคลัง แล้วบันทึกผลว่าตรงกันหรือไม่

**เหมาะกับใคร:** พนักงานที่ทำการตรวจนับประจำวันหรือประจำสัปดาห์

**ทำงานอย่างไร:**

- สินค้าแต่ละชนิดตั้งได้ว่าจะนับแบบไหน:
  - **นับตัวเลขจริง** — ต้องกรอกจำนวนที่นับได้ ผลการนับมีผลต่อคะแนนความแม่นยำ
  - **เช็คแค่ว่าพอใช้** — ตอบว่าพอหรือไม่พอ ไม่บังคับนับตัวเลข ไม่เอาไปคิดคะแนนความแม่นยำ
- ทุกครั้งที่ตรวจนับ ระบบเก็บประวัติไว้ — ดูย้อนหลังได้ว่าใครนับ เมื่อไหร่ ตรงกับในคลังหรือไม่

**จุดเด่น:** ไม่บังคับนับทุกอย่างแบบเดียวกัน — เมล็ดกาแฟต้องนับจริง แต่ถุงกระดาษอาจแค่เช็คว่ายังพอใช้ ช่วยให้การนับสต็อกเร็วขึ้นโดยไม่เสียความแม่นยำในสินค้าสำคัญ

#### Inventory Accuracy

`/[locale]/inventory/accuracy` · รายงานความแม่นยำ

**ใช้ทำอะไร:** แสดงคะแนนความแม่นยำของการตรวจนับสต็อก — เฉพาะสินค้าที่ตั้งให้ **นับตัวเลขจริง** เท่านั้น มีกราฟวัดให้เห็นภาพรวม

**เหมาะกับใคร:** ผู้จัดการที่ต้องการติดตามว่าทีมนับสต็อกแม่นแค่ไหน และสินค้าไหนมักนับคลาดเคลื่อน

**ทำงานอย่างไร:**

- คำนวณจากประวัติการตรวจนับทั้งหมด
- ไม่เอาสินค้าแบบ "เช็คพอใช้" มาคิดในคะแนน — เพราะไม่ได้นับตัวเลขจริง
- แสดงเป็นเปอร์เซ็นต์และกราฟวัด อ่านง่ายในที่ทำงาน

**จุดเด่น:** ตัวเลขความแม่นยำสะท้อนสินค้าที่สำคัญจริงๆ ไม่ถูกบิดเพราะรายการที่แค่เช็คว่าพอใช้

---

#### Branch Withdraw

`/[locale]/inventory/branch-withdraw` · เบิกของไปสาขา 2

**ใช้ทำอะไร:** บันทึกการเบิกสินค้าจากคลังหลักไป **สาขา 2** — เลือกหลายรายการแล้วบันทึกครั้งเดียว พร้อมดูประวัติการเบิกย้อนหลัง

**เหมาะกับใคร:** ผู้จัดการคลังหรือคนที่รับผิดชอบส่งของไปสาขาย่อย

**ทำงานอย่างไร:**

- เลือกสินค้าและจำนวนที่เบิก แล้วบันทึกเป็นชุดเดียว — สต็อกคลังหลักลดทันที
- ถ้ายังกรอกไม่เสร็จ ระบบเก็บแบบร่างไว้ก่อน — กลับมาบันทึกต่อได้ ไม่หาย
- เมื่อเบิกเสร็จ อุปกรณ์อื่นที่เปิดระบบคลังอยู่จะได้รับแจ้งเตือน
- มีประวัติทุกครั้งที่เบิก — ตรวจสอบย้อนหลังได้

**จุดเด่น:** ไม่ต้องจดใน Excel หรือกระดาษแล้วค่อยมาคีย์ทีหลัง — สต็อกตรงทันที มีหลักฐานครบ

### 📊 ข้อมูลร้าน

#### Sales

`/[locale]/sales` · วิเคราะห์ยอดขาย

**ใช้ทำอะไร:** อัปโหลดไฟล์ Excel ยอดขายจากเครื่อง POS (ระบบขายหน้าร้าน) แล้วดูสรุปยอดขายตามหมวดสินค้าและช่วงเวลา

**เหมาะกับใคร:** เจ้าของร้าน ผู้จัดการ หรือคนที่ต้องการดูแนวโน้มยอดขายโดยไม่ต้องนั่งไล่ตัวเลขใน Excel เอง

**ทำงานอย่างไร:**

- อัปโหลดไฟล์ Excel ที่ export จาก POS — ระบบอ่านและเก็บข้อมูลในฐานข้อมูล
- จัดหมวดสินค้าได้ (เช่น กาแฟ เบเกอรี เครื่องดื่มอื่น)
- ดูยอดรวมตามช่วงวันที่ที่อัปโหลดมา

**จุดเด่น:** ไม่ต้องเชื่อมต่อ API กับ POS โดยตรง (ซึ่งมักยุ่งยากและมีค่าใช้จ่าย) — แค่อัปโหลดไฟล์ที่มีอยู่แล้วก็วิเคราะห์ได้

---

#### Maintenance

`/[locale]/maintenance` · บันทึกการซ่อม

**ใช้ทำอะไร:** บันทึกประวัติการซ่อมบำรุง — เครื่องชงกาแฟ อุปกรณ์ต่างๆ วันที่ซ่อม อาการ ค่าใช้จ่าย และผู้รับผิดชอบ

**เหมาะกับใคร:** ผู้จัดการร้าน ช่างซ่อม หรือคนที่ดูแลอุปกรณ์ในร้าน

**ทำงานอย่างไร:**

- กรอกข้อมูลในตารางแบบเดียวกับหน้าคลัง — ใช้คุ้นมือถ้าเคยใช้ Inventory แล้ว
- ข้อมูลเก็บในฐานข้อมูลปลอดภัย ค้นหาย้อนหลังได้
- ผู้ช่วย AI (บรู) อ่านประวัติซ่อมได้ — ถามว่า "เครื่องชงซ่อมล่าสุดเมื่อไหร่" ได้

**จุดเด่น:** ประวัติซ่อมอยู่ในระบบเดียวกับตารางงานและคลัง — ไม่ต้องเปิดสมุดจดหรือไลน์แยก

### 🤖 ระบบและ AI

#### Settings

`/[locale]/settings` · ตั้งค่าและความปลอดภัย

**ใช้ทำอะไร:** ศูนย์ตั้งค่าทั้งหมด — เปลี่ยนธีมสี ดูประวัติการเข้าใช้ระบบ ลงทะเบียนลายนิ้วมือหรือใบหน้า (เพื่อเข้าเร็วขึ้น) จัดการการแจ้งเตือน และตัดการเชื่อมต่อของอุปกรณ์อื่นที่ยังล็อกอินอยู่

**เหมาะกับใคร:** ผู้ดูแลระบบ หรือผู้จัดการที่ต้องการควบคุมว่าใครเข้าใช้และจากอุปกรณ์ไหน

**ทำงานอย่างไร:**

- **ธีม:** เลือกโหมดสว่าง มืด หรือตามการตั้งค่าของเครื่อง
- **ประวัติเข้าใช้:** บันทึกทุกครั้งที่มีคนเข้าระบบ พร้อมระบุอุปกรณ์และเวลา
- **ลายนิ้วมือ/ใบหน้า:** หลังใส่ PIN ครั้งแรก สามารถลงทะเบียนเครื่องที่เชื่อถือได้ — ครั้งถัดไปเข้าได้เร็วขึ้น
- **ตัดการเชื่อมต่อระยะไกล:** บังคับให้อุปกรณ์อื่นออกจากระบบได้ (เช่น มือถือที่ลืมล็อกเอาต์)
- **การแจ้งเตือน:** เปิด-ปิดประเภทการแจ้งเตือนที่ต้องการ

**จุดเด่น:** ไม่ใช่แค่หน้าเปลี่ยนสี — เป็นศูนย์ควบคุมความปลอดภัยสำหรับทีมเล็กที่ใช้ PIN ร่วมกัน แต่ยังต้องการรู้ว่าใครเข้าเมื่อไหร่

#### AI Chat — บรู

เปิดได้ทุกหน้า · ผู้ช่วย AI ของร้าน

**ใช้ทำอะไร:** ถามคำถามเกี่ยวกับร้านเป็นภาษาพูดธรรมดา — เช่น "วันนี้ใครอยู่กะเช้า" "สต็อกนมเหลือน้อยไหม" "ยอดขายสัปดาห์นี้เท่าไหร่" "พรุ่งนี้หยุดไหม" — บรูจะดึงข้อมูลจริงจากระบบมาตอบ

**เหมาะกับใคร:** ผู้จัดการและพนักงานที่มีสิทธิ์แก้ไข (PIN แบบเต็มสิทธิ์) — โหมดดูอย่างเดียวใช้บรูไม่ได้เพื่อความปลอดภัย

**ทำงานอย่างไร:**

- ใช้ AI รุ่น Gemini เป็นสมองหลัก
- อ่านข้อมูลร้านผ่านช่องทางเดียวที่ควบคุมได้ — ป้องกันการดึงข้อมูลส่วนที่ไม่ควรเปิดเผย
- ตอบทีละส่วนแบบพิมพ์สดๆ (streaming) — เห็นคำตอบค่อยๆ โผล่มา
- กรองคำตอบเพื่อความปลอดภัยก่อนแสดงบนหน้าจอ
- สามารถค้นหาข้อมูลบนอินเทอร์เน็ตเสริมได้ในบางคำถาม (เช่น ข่าวหรือข้อมูลทั่วไป)

**จุดเด่น:** แชทบอททั่วไปตอบจากความรู้ทั่วไปที่อาจไม่ตรงกับร้านคุณ — บรูตอบจากข้อมูลร้านจริงที่อัปเดตอยู่ จึงน่าเชื่อถือกว่าเมื่อต้องตัดสินใจเรื่องงาน

### 🔐 ความสามารถของแพลตฟอร์ม

ฟีเจอร์พื้นฐานที่ทำให้ระบบใช้งานบนมือถือในร้านได้จริง ไม่จำกัดแค่คอมพิวเตอร์ตั้งโต๊ะ

| ฟีเจอร์ | ใช้ทำอะไร | รายละเอียดเพิ่ม |
| --- | --- | --- |
| **PIN Gateway** | ใส่ PIN 6 หลักก่อนเข้าแอป | มี 2 ระดับ: แก้ไขได้ทุกอย่าง หรือดูอย่างเดียว |
| **Passkeys** | เข้าด้วยลายนิ้วมือหรือใบหน้า | ลงทะเบียนหลังใส่ PIN ครั้งแรกบนเครื่องนั้น |
| **Web Push** | แจ้งเตือนบนมือถือ/แท็บเล็ต | เช่น สต็อกถูกแก้ หรือรายงานกะประจำวัน |
| **PWA** | ติดตั้งเป็นแอปบนหน้าจอโฮม | เปิดเร็วเหมือนแอปจริง ใช้ offline บางส่วนได้ |
| **ธีมสว่าง/มืด** | เลือกโทนสีตามชอบ | การ์ดกะยังอ่านง่ายทั้งสองโหมด |
| **สองภาษา** | ไทย (หลัก) และอังกฤษ | สลับได้ทั้งระบบ ผ่าน `/th` หรือ `/en` |

#### ระบบความปลอดภัย

```text
ใส่ PIN → ระบบจำการเข้าใช้อย่างปลอดภัย → เข้าถึงฐานข้อมูลตามสิทธิ์ที่กำหนด
→ ทุกครั้งที่แก้ข้อมูล ระบบตรวจสิทธิ์ก่อนเสมอ
```

**ทำไมถึงออกแบบแบบนี้:** ร้านเล็กมักไม่อยากสร้างบัญชีผู้ใช้รายคน (ยุ่งยาก พนักงานลืมรหัส) — ระบบนี้ใช้ PIN ร่วมกันได้ แต่ยังมีประวัติว่าใครเข้าเมื่อไหร่ ตัดการเชื่อมต่ออุปกรณ์ที่สูญหายได้ และจำกัดสิทธิ์ว่าใครแก้ข้อมูลได้บ้าง

## 🚀 Installation

ส่วนนี้สำหรับนักพัฒนาหรือผู้ดูแลระบบที่ต้องการรันโปรเจกต์บนเครื่องตัวเอง หรือเตรียม deploy ขึ้นเซิร์ฟเวอร์

### สิ่งที่ต้องมี

| รายการ | รายละเอียด |
| --- | --- |
| **Node.js 24** | โปรแกรมรันฝั่งเซิร์ฟเวอร์ของเว็บแอป — ดาวน์โหลดได้ที่ [nodejs.org](https://nodejs.org) (Vercel ใช้ `24.x`) |
| **npm** | ติดตั้งมาพร้อม Node.js ใช้สำหรับดาวน์โหลดและจัดการแพ็กเกจ |
| **บัญชี Supabase** | บริการฐานข้อมูลบนคลาวด์ — สร้างโปรเจกต์ใหม่แล้วนำ URL และ API keys มาใส่ในไฟล์ตั้งค่า |
| **Git** | สำหรับโคลนโค้ดจาก repository (ถ้ายังไม่มีโค้ดในเครื่อง) |

### ขั้นตอน

```bash
# 1. โคลนโปรเจกต์ (แทน <repo-url> ด้วย URL จริงของ repository)
git clone <repo-url>
cd black-and-brew

# 2. ติดตั้งแพ็กเกจที่โปรเจกต์ต้องใช้
npm install

# 3. สร้างไฟล์ตั้งค่าจากตัวอย่าง
cp .env.example .env.local
# เปิด .env.local แล้วกรอกค่าตามหัวข้อ Environment Variables ด้านล่าง

# 4. รันเซิร์ฟเวอร์สำหรับพัฒนา
npm run dev
```

**หลังรันสำเร็จ:** เปิด [http://localhost:3000](http://localhost:3000) ในเบราว์เซอร์ — ระบบจะพาไปหน้า `/th` และขอให้ใส่ PIN (ค่า PIN มาจาก `APP_PIN` ในไฟล์ `.env.local`)

**ถ้าเจอปัญหา:**

- หน้าเว็บขึ้น error เรื่อง Supabase → ตรวจว่ากรอก `NEXT_PUBLIC_SUPABASE_URL` และ `NEXT_PUBLIC_SUPABASE_ANON_KEY` ถูกต้อง
- ใส่ PIN แล้วเข้าไม่ได้ → ตรวจว่า `APP_PIN` ใน `.env.local` ตรงกับที่ใส่ (6 หลัก)
- พอร์ต 3000 ถูกใช้อยู่ → ปิดโปรแกรมอื่นที่ใช้พอร์ตนี้ หรือรัน `npm run dev -- -p 3001` เพื่อใช้พอร์ตอื่น

### 📜 คำสั่งที่ใช้บ่อย

## 🔑 Environment Variables

ตัวแปรเหล่านี้เป็น "รหัสและการตั้งค่า" ที่บอกให้แอปรู้ว่าจะเชื่อมต่อฐานข้อมูลที่ไหน ใช้ PIN อะไร และเรียก AI ได้หรือไม่

**วิธีตั้งค่า:**

1. คัดลอกไฟล์ [.env.example](.env.example) ไปเป็น `.env.local` (สำหรับรันบนเครื่องตัวเอง)
2. เปิด `.env.local` แล้วกรอกค่าจริงแทนข้อความ placeholder
3. สำหรับเว็บที่ขึ้น production บน Vercel → ตั้งค่าเดียวกันใน **Vercel Dashboard** → Settings → Environment Variables

> **คำอธิบายประเภท:** `PUBLIC` = ค่าที่เบราว์เซอร์เห็นได้ (ปลอดภัยต่อการเปิดเผย) · `SECRET` = ค่าลับ เก็บฝั่งเซิร์ฟเวอร์เท่านั้น ห้ามแชร์ · `OPTION` = ไม่บังคับ แต่ถ้าไม่ใส่ฟีเจอร์นั้นจะใช้ไม่ได้

### 🗄 Supabase (จำเป็น — ไม่มีแล้วระบบรันไม่ได้)

| ตัวแปร | ประเภท | อธิบาย |
| --- | --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | PUBLIC | ที่อยู่โปรเจกต์ Supabase ของคุณ (หน้า Settings → API ใน Supabase Dashboard) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | PUBLIC | คีย์สำหรับแอปฝั่งผู้ใช้ — ใช้ร่วมกับระบบสิทธิ์ในฐานข้อมูล |
| `SUPABASE_SERVICE_ROLE_KEY` | SECRET | คีย์สำหรับงานหลังบ้าน (AI, รายงาน) — **ห้าม**ใส่คำว่า `NEXT_PUBLIC_` นำหน้า |

### 🔒 ล็อกอินและตำแหน่งร้าน

| `WEBAUTHN_ORIGIN` | SECRET, OPTION | URL เต็มของเว็บสำหรับลายนิ้วมือ/ใบหน้าบน production |

## 🏗 Architecture

โครงสร้างโฟลเดอร์หลักของโปรเจกต์ — ช่วยให้รู้ว่าโค้ดส่วนไหนอยู่ที่ไหน

```text
src/
├── app/
│   ├── page.tsx              # หน้าแรก — พาไป /th
│   ├── manifest.ts           # ตั้งค่า PWA (ติดตั้งเป็นแอป)
│   ├── actions/              # ฟังก์ชันแก้ข้อมูล (คลัง, ตารางงาน, ล็อกอิน, ยอดขาย, …)
│   ├── api/                  # ช่องทางพิเศษ (แชท AI, รายงานประจำวัน, แจ้งเตือน, offline)
│   └── [locale]/             # หน้าเว็บทั้งหมด (แยกภาษาไทย/อังกฤษ)
│       ├── <feature>/page.tsx, *Client.tsx
│       └── <feature>/_components/   # ส่วนประกอบเฉพาะแต่ละฟีเจอร์
├── components/               # ส่วนประกอบ UI ที่ใช้ร่วมกันหลายหน้า
├── lib/                      # โค้ดช่วย (เชื่อมฐานข้อมูล, คำนวณคลัง, สีกะ, …)
├── i18n/                     # ระบบสองภาษา
└── proxy.ts                  # ตัวกลางจัดการภาษาและเส้นทาง URL
```

### จุดสำคัญด้านเทคนิค

- 🗃 **ฐานข้อมูล** — ใช้ Supabase (PostgreSQL) โฮสต์ใกล้ไทย ข้อมูลตอบสนองเร็ว
- 🔄 **อัปเดตสต็อก** — ทุกการแก้สต็อกผ่านฟังก์ชันพิเศษในฐานข้อมูลที่ล็อกข้อมูลก่อนแก้ ป้องกันข้อมูลชนกัน
- 📋 **นโยบายนับสินค้า** — แยกสินค้าที่ต้องนับตัวเลขจริง กับสินค้าที่แค่เช็คว่าพอใช้
- 🛡 **สิทธิ์ข้อมูล** — เฉพาะคนที่ล็อกอินแล้วเท่านั้นที่อ่านหรือแก้ข้อมูลได้

## 🤝 Contributing

สำหรับทีมที่ต้องการแก้ไขหรือเพิ่มฟีเจอร์ในโปรเจกต์

1. **อ่านเอกสารก่อน** — [docs/SOP.md](docs/SOP.md) (ขั้นตอนพัฒนา) และ [docs/design.md](docs/design.md) (มาตรฐานหน้าตา)
