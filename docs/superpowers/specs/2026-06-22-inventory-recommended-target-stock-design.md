# Inventory Recommended Target Stock Design

## Purpose

คลังสินค้าจะยังคงใช้ `จำนวนที่ต้องมี` เดิมเป็นค่าหลักที่เจ้าของร้านควบคุมเอง แต่ระบบจะช่วยคำนวณ `ค่าที่แนะนำ` เพื่อให้สต็อกรองรับการใช้งานอย่างน้อย 2 อาทิตย์ โดยคำนึงถึงประวัติการใช้จริง วันหยุด lead time และความเสี่ยงของขาดตลาด

ฟีเจอร์นี้เป็นแบบ assisted decision: ระบบเสนอค่า เจ้าของร้านเลือกใช้เอง ไม่อัปเดต `จำนวนที่ต้องมี` อัตโนมัติ

## Goals

- แสดงค่าเดิมและค่าที่แนะนำในช่อง `จำนวนที่ต้องมี` ช่องเดิมเท่านั้น
- ไม่เพิ่มคอลัมน์ใหม่ และไม่เปลี่ยนขนาด/โครงสร้างตารางที่ใช้ได้ดีอยู่แล้วทั้งมือถือและเดสก์ท็อป
- ให้เจ้าของร้านตั้ง `ความเสี่ยงของขาดตลาด` และ `lead time` ต่อสินค้าได้เอง
- ให้เจ้าของร้านกดใช้ค่าที่แนะนำต่อรายการได้
- อธิบายเหตุผลของค่าที่แนะนำได้แบบตรวจสอบย้อนกลับ ไม่เป็นกล่องดำ

## Non-Goals

- ไม่ให้ AI หรือระบบเปลี่ยน `จำนวนที่ต้องมี` อัตโนมัติ
- ไม่เพิ่มคอลัมน์ในตาราง inventory
- ไม่เปลี่ยน layout ตาราง inventory บนมือถือหรือเดสก์ท็อป
- ไม่เพิ่มระดับความเสี่ยง `วิกฤต`
- ไม่นำธุรกรรมประเภทอื่นมาเป็นเงื่อนไขพิเศษในสูตร ยกเว้นการกรอง `OUT` ที่ใหญ่ผิดปกติ

## UI Design

ช่อง `จำนวนที่ต้องมี` แสดงแบบกระชับ:

```text
20 → 28
```

- `20` คือ `target_stock` ปัจจุบัน
- `28` คือค่าที่ระบบแนะนำ
- ถ้าค่าที่แนะนำไม่สูงกว่าค่าเดิม ให้แสดงเฉพาะค่าเดิม เช่น `20`
- ถ้าค่าที่แนะนำสูงกว่าเดิมเล็กน้อยจนไม่มีนัยสำคัญ ให้ซ่อนลูกศรเพื่อลดความรก

เกณฑ์แสดงลูกศร:

```text
แสดงเมื่อ recommended_target_stock > target_stock
และต่างกันอย่างน้อย 10% หรืออย่างน้อย 2 หน่วย
```

แต่ละรายการมีปุ่มเล็กใน row เดิมสำหรับเปิดรายละเอียด โดยต้องไม่ทำให้ตารางกว้างขึ้น สูงขึ้น หรือเพิ่มคอลัมน์ใหม่ ปุ่มนี้ใช้สำหรับ:

- แก้ระดับความเสี่ยงของขาดตลาด
- แก้ lead time
- ดูเหตุผลของค่าที่แนะนำ
- กดใช้ค่าที่แนะนำกับรายการนั้น

บนเดสก์ท็อปควรเปิดเป็น popover ขนาดเล็ก บนมือถือควรเปิดเป็น bottom sheet เพื่อรักษาความอ่านง่ายและ touch target

## Risk Levels

ระดับความเสี่ยงกำหนดเองต่อสินค้า:

- `ปกติ`: ไม่เพิ่ม buffer ของขาดตลาด
- `ปานกลาง`: เพิ่ม buffer 15%
- `สูง`: เพิ่ม buffer 30%

ระดับนี้ใช้กับสินค้าที่รู้ว่าซื้อยาก ขาดตลาดง่าย หรือ supplier ส่งไม่แน่นอน โดยช่วงแรกให้เจ้าของร้านตั้งเองก่อน

## Lead Time

`lead_time_days` คือจำนวนวันที่ต้องรอหลังสั่งซื้อจนได้ของ

ค่าเริ่มต้นแนะนำ:

- ของซื้อใกล้ร้านหรือหาง่าย: 1 วัน
- supplier ปกติ: 3 วัน
- ของเฉพาะทางหรือส่งไม่แน่นอน: 7 วัน

เจ้าของร้านแก้ต่อสินค้าได้จากปุ่มเล็กในรายการ

## Calculation

สูตรหลักไม่ใช้ AI เป็นตัวคำนวณตัวเลขหลัก เพื่อให้ตรวจสอบได้:

```text
recommended_target_stock =
  base_usage_14_days
  + holiday_buffer
  + lead_time_buffer
  + shortage_risk_buffer
```

### Base Usage

สำหรับสินค้าที่มีประวัติ `OUT`:

```text
average_daily_usage = weighted average ของ OUT ย้อนหลัง
base_usage_14_days = average_daily_usage × 14
```

ควรใช้ช่วงข้อมูลย้อนหลังอย่างน้อย 30-90 วัน ถ้ามีข้อมูลพอ โดยให้น้ำหนักข้อมูลล่าสุดมากกว่าข้อมูลเก่าเล็กน้อย

สำหรับสินค้าประเภทไม่ต้องเบิกหรือมีประวัติ `OUT` ไม่พอ:

- ใช้ข้อมูลรับเข้าและรอบการเติมเป็นสัญญาณรอง
- ถ้ามีข้อมูลน้อย ให้แสดง confidence ว่า `ข้อมูลน้อย`
- ไม่ควรเสนอค่าที่ก้าวร้าวเกินไปจากข้อมูลไม่พอ

### Abnormal OUT Handling

กันเฉพาะ `OUT` ที่ใหญ่ผิดปกติออกจากสูตรหรือให้ลดน้ำหนัก เพราะอาจเป็นการเบิกพิเศษหรือเหตุการณ์ที่ไม่ควรแทน usage ปกติ

แนวทางเริ่มต้น:

```text
ถ้า OUT ครั้งหนึ่งมากกว่า median OUT ของสินค้านั้น 3 เท่า
หรือมากกว่า percentile 95 ของ OUT ย้อนหลัง
ให้ถือว่าเป็น abnormal OUT
```

รายการ abnormal OUT ไม่ควรถูกนับเต็มใน `average_daily_usage` แต่ยังควรแสดงในเหตุผลหรือ diagnostic ได้ว่าเคยมีรายการผิดปกติ

### Holiday Buffer

ใช้ข้อมูลวันหยุดจากตาราง `holidays` ที่ระบบมีอยู่แล้ว

```text
holiday_multiplier = 1 + (จำนวนวันหยุดใน 14 วันข้างหน้า × 0.10)
```

ถ้าเจอช่วงหยุดติดกัน 3 วันขึ้นไป ให้เพิ่ม long-weekend buffer:

```text
long_weekend_extra = 0.20
```

สูตร:

```text
holiday_buffer = base_usage_14_days × (holiday_multiplier + long_weekend_extra - 1)
```

### Lead Time Buffer

```text
lead_time_buffer = average_daily_usage × lead_time_days
```

### Shortage Risk Buffer

```text
ปกติ = base_usage_14_days × 0
ปานกลาง = base_usage_14_days × 0.15
สูง = base_usage_14_days × 0.30
```

### Rounding

ค่าที่แนะนำควรปัดขึ้นเป็นจำนวนเต็มเสมอ:

```text
recommended_target_stock = ceil(calculated_value)
```

## Confidence

แสดงระดับความมั่นใจเฉพาะใน popover/bottom sheet ไม่แสดงบนตารางหลัก

ตัวอย่าง:

- `มั่นใจสูง`: มี OUT สม่ำเสมอและมีข้อมูลย้อนหลังพอ
- `ข้อมูลปานกลาง`: มีข้อมูลบางส่วน แต่ช่วงเวลายังสั้นหรือ usage แกว่ง
- `ข้อมูลน้อย`: ไม่มี OUT หรือมีข้อมูลไม่พอ ต้องใช้ประวัติรับเข้า/การตั้งค่าด้วยมือประกอบ

## Explanation

popover/bottom sheet ต้องมีส่วน `ดูเหตุผล` แบบสั้น เช่น:

```text
ใช้เฉลี่ย 1.4/วัน
14 วัน = 20
lead time 3 วัน = +5
วันหยุด = +3
ความเสี่ยงสูง = +6
แนะนำ 34
```

เหตุผลนี้ช่วยให้เจ้าของร้านเชื่อถือค่าที่แนะนำก่อนกดใช้

## Apply Recommended Value

ปุ่ม `ใช้ค่าที่แนะนำ` ทำงานต่อรายการเท่านั้นในระยะแรก

เมื่อกด:

- อัปเดต `target_stock` เป็นค่าที่แนะนำ
- คงการ autosave / Supabase sync ตาม pattern เดิมของ inventory
- หลังอัปเดตสำเร็จ ช่องควรกลับมาแสดงค่าเดี่ยว เช่น `28`
- ควรบันทึกเหตุผลหรือ metadata ได้ถ้าระบบ audit มีรองรับ

## AI Role

AI ไม่ควรเป็นตัวคำนวณเลขหลักในระยะแรก เพราะสูตร stock ต้องตรวจสอบซ้ำได้

AI สามารถเพิ่มในระยะถัดไปเพื่อ:

- อธิบายเหตุผลเป็นภาษาคน
- ช่วยจับ pattern ที่ผิดปกติ
- ช่วยแนะนำว่าสินค้าไหนควรถูกตั้งเป็น `ปานกลาง` หรือ `สูง`
- ช่วยเชื่อมโยงสินค้าหน้าบาร์กับยอดขายเมนู เมื่อข้อมูลพร้อม

ตัวเลขหลักยังควรมาจาก deterministic formula ก่อน

## Testing And Validation

ควรมี test สำหรับ:

- แสดง `20 → 28` เมื่อค่าที่แนะนำสูงกว่าเดิมตาม threshold
- แสดง `20` เมื่อค่าที่แนะนำไม่สูงกว่าเดิมหรือสูงขึ้นเล็กน้อยเกินไป
- คำนวณ buffer วันหยุดจากจำนวนวันหยุดใน 14 วันข้างหน้า
- คำนวณ lead time buffer จาก `average_daily_usage × lead_time_days`
- คำนวณ shortage risk buffer สำหรับ `ปกติ`, `ปานกลาง`, `สูง`
- กรองหรือลดน้ำหนัก `OUT` ที่ใหญ่ผิดปกติ
- สินค้าข้อมูลน้อยต้องแสดง confidence เป็น `ข้อมูลน้อย`
- ปุ่มรายละเอียดไม่เปลี่ยนขนาด/จำนวนคอลัมน์ของตาราง inventory

## Implementation Defaults

- ปุ่มเล็กควรอยู่ภายใน cell เดิมของ `จำนวนที่ต้องมี` เพื่อให้ context ชัด แต่ต้องเป็น overlay/inline control ที่ไม่เพิ่มความกว้างหรือความสูงของ row
- เก็บ `shortage_risk` และ `lead_time_days` ใน `inventory_items` โดยตรง เพื่อให้แก้ต่อรายการและ sync แบบ realtime ได้ง่าย
- ใช้ช่วงย้อนหลังเริ่มต้น 60 วันสำหรับ `average_daily_usage`
- ถ้าข้อมูลย้อนหลังน้อยกว่า 14 วัน ให้คำนวณได้แต่ต้องแสดง confidence เป็น `ข้อมูลน้อย`
